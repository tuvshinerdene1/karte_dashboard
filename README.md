```
git clone
```

```
ncp .env.example .env
```
Edit `.env` with your configuration:
```
PORT=8080
NODE_ENV=development
DB_HOST=localhost
DB_PORT=5432
DB_NAME=hospital_db
DB_USER=postgres
DB_PASSWORD=1234
JWT_SECRET=your_secret_key_here
```

## Set up PostgreSQL database
**Create the database:**
```
createdb hospital_db
```
Or you can use PG admin to create your db.(DB settings must match the .env above.)

**Run automated migrations to build out all tables and relationships:**
```
npm run migrate:up
```

**To drop all tables :**
```
npm run migrate:down
```
**Populating tables with mock data**

there is ready to use sql query file in dashboard-backend/queries.
you can directly run that query in pgadmin.
## Tables and their explanation
### Core configuration tables

these tables define the static structure of the hospital system

**1.hospitals**

Stores information about each hospital facility.

* id: UUID (Primary key)
* name: Name of the hospital
* location: physical area or city district

**2.services**

Stores global medical service categories

* id: UUID
* service_name: Name of the service

**3.service_steps**

A "Template" table. It defines the generic steps a patient takes for a specific service.

* id: UUID (Primary Key).

* service_id: Reference to the parent service.

* step_name: Name of the step (e.g., "Triage", "X-Ray").

* step_order: Integer representing the sequence (1, 2, 3...).

**4.hospital_steps**

Critical Configuration Table. This links a generic service_step to a specific hospital and sets the timing rules for bottleneck detection.

* id: UUID (Primary Key).

* hospital_id: Reference to the hospital.

* step_id: Reference to the generic service_step.

* mid_threshold_minutes: The "Warning" threshold. Exceeding this marks the process as slow.

* high_threshold_minutes: The "Critical" threshold. Exceeding this marks a severe bottleneck.

### 2. People & Roles
**1.users**

System-level accounts for authentication.

* id: UUID (Primary Key).

* username: Unique login name.

* password_hash: Securely hashed password.

* role: User permissions (admin, operator, hospital_staff).

**2.staff**

The medical professionals working in the hospitals.

* id: UUID (Primary Key).

* user_id: Reference to the login account (Optional).

* hospital_id: The hospital where the staff member is based.

* full_name: Doctor/Staff name.

* specialization: Medical field.

**3.staff_step_assignments**

Maps which staff members are authorized/assigned to perform which hospital_steps.

* id: UUID (Primary Key).

* staff_id: Reference to the staff member.

* hospital_step_id: Reference to the specific hospital step.

### Transactional Data (The Flow)
**visits**

Represents a single patient's total journey from check-in to discharge.

* id: UUID (Primary Key).

* hospital_id / service_id: Where and what service the patient is receiving.

* patient_identifier: A string ID for the patient (e.g., "PAT-101").

* start_time: When the journey began.

* status: Global visit status (active, completed, bottlenecked).

**visit_events**

The "Heart" of the system. Tracks the patient as they move through specific steps. It uses a 3-timestamp logic to calculate wait times.

* id: UUID (Primary Key).

* visit_id: Reference to the parent visit.

* hospital_step_id: The specific step currently occurring.

* staff_id: The doctor/staff assigned to this event (NULL if the patient is still waiting).

* status: The current state of the patient in this step:

* waiting: Patient has arrived but no staff is available.

* in_progress: Patient is inside with a staff member.

* completed: The step is finished.

* arrived_at: Timestamp when the patient joined the queue for this step.

* entered_at: Timestamp when the patient actually entered the room and started the process.

* exited_at: Timestamp when the patient finished the step.

* is_ml_flagged: Boolean for future Machine Learning integration.

### Operational Tables
**notifications**

Alerts generated by the system when bottlenecks are detected.

* id: UUID (Primary Key).

* user_id: Who the notification is for.

* visit_event_id: Link to the specific event that triggered the alert.

* type: Severity level (warning, critical).

* message: Text description of the bottleneck.

* is_read: Boolean read status.

**support_requests**

A ticketing system for staff to report issues or request help from system operators.

* id: UUID (Primary Key).

* hospital_id: Where the request originated.

* requester_staff_id: The staff member asking for help.

* assigned_operator_id: The admin/operator handling the ticket.

* title / description: Details of the issue.

* status: open, in_progress, resolved.

* priority: low, medium, high.

## Flow simulator (work in progress)
This Python-based simulator is designed to stress-test our Hospital Bottleneck Detection system. 
It mimics patient movement through various hospital services (Cardiology, ER, etc.) by sending real-time events to our Node.js/Express API.

### Features
1. Traffic Generation: Automatically creates patient journeys based on database metadata.
2. System Stress Testing: Adjustable multipliers to simulate slow processing times.
3. Simulated Speed: Control the ratio between real-time and simulation-time.
4. Concurrent Processing: Handles multiple patients simultaneously using threading.

### Set up instructions 
**1. Create the virtual environment**
```
python -m venv vhospital
```

**2. activate the environment**

Windows (Command Prompt)
```	
vhospital\Scripts\activate
```

Windows (PowerShell)	
```	
.\vhospital\Scripts\Activate.ps1
```

macOS / Linux	
```		
source vhospital/bin/activate
```

**3. install dependencies **

```
pip install -r requirements.txt
```

### Running simulator
1. ensure the backend is running
2. launch the streamlit 
```
streamlit run simulator.py
```

### Deactivating the environment
```
deactivate
```


### API endpoints required 
```
GET /api/hospitals - Fetch list of hospitals.

GET /api/hospitals/:id/services - Fetch services for a hospital.

GET /api/hospitals/:id/services/:name/steps - Fetch the ordered steps for a specific service.

POST /api/events - To send START and END signals for patients.
```
